Why zero-dependency vanilla?

The app must open from a file share or email attachment inside a client network with no build step, no CDN reach and no install rights. Every capability — parsing, aggregation, retrieval, animation — is implemented in plain browser APIs, so the one HTML file is the deployment.

Heat map calculation logic

Records→Bucket key→Count grid→Normalise→Gamma curve→Greyscale shade→Contrast flip

1 · Bucketing

Every record is mapped to a cell by the pair (category, columnKey). The column key depends on the selected dimension:

month    → opened.getUTCFullYear() + "-" + (opened.getUTCMonth()+1)   // "2025-11"
region   → record.region                                              // "Gauteng"
priority → record.priority                                            // "P1".."P4"

2 · Aggregation

A single pass builds three structures: per-category totals (used to sort rows hottest-first), the cell grid grid[cat + "||" + colKey] += 1, and the running maximum max of any cell — the normalisation anchor.

3 · Intensity with a gamma curve

t = (value / max) ^ 0.6        // gamma < 1 lifts mid-range differences
grey = round(244 − t × 234)    // 244 (near-white) … 10 (ink black)
background = rgb(grey, grey, grey)
text = t > 0.45 ? white : ink  // contrast flip for readability

Raw counts follow a long-tail distribution: one festive-season spike would otherwise wash every other cell to near-white. The 0.6 exponent compresses the top and stretches the middle, so a cell at 30% of max still reads visibly darker than one at 10%.

4 · Zero cells & the total column

Empty cells render a fixed bone tone (#f4f3ef) with a middle-dot placeholder — visually silent so genuine heat stands out. The Total column sits outside the shading scheme (white, black rule) as the row anchor.

Agent interaction logic

Deterministic local retrieval — no API, no keys
Query→ID detect→Tokenise→Score SOPs→Threshold gate→Data signal→RCA · Fix · Mitigation

1 · Record-ID detection

The query is first scanned with /\b(RET-\d+|INC\d{5,}|PRB\d{4,}|CHG\d{4,})\b/. On a hit, the record is looked up in the unified store and its summary and category text are appended to the query before scoring — so "what happened with INC0010042" routes to the right SOP even though the question itself carries no symptom words.

2 · Weighted SOP scoring

SignalMatch againstWeightRationale
Curated keywordsRaw query substring+4 eachAuthor-chosen phrases ("code 91", "3-d secure") are the strongest intent signal
Title wordsQuery tokens+2 eachTitles name the failure mode
Service-area wordsQuery tokens+2 eachCatches "POS", "loyalty", "EDI" style questions
RCA body wordsQuery tokens+0.4 eachWeak corroboration only — long text must not dominate

Tokens are lowercase, alphanumeric, length > 2. SOPs are ranked by score; the winner must clear the threshold of 3. Score ≥ 8 is labelled high confidence, otherwise medium — printed on every answer's source line.

3 · Honest fallback

Below threshold, the agent refuses to guess. It says it can't match, and coaches the user: name a service area or paste a record ID. A wrong SOP presented confidently is worse than no SOP.

4 · Data signal (grounding in your data)

The matched SOP's service area is joined back to the loaded records to compute: total matching records, open count, mean resolve hours over resolved records, top region, and month-over-month trend (May → Jun 2026 in the demo window). The SOP tells you what to do; the data signal tells you how big and how urgent.

5 · Answer contract

Every successful answer renders the same five blocks, in order: matched record header (if an ID was given) → Root cause analysis → Fix now (ordered steps) → Mitigate for the future → data-signal chips → source line (SOP ID · area · confidence). Consistency is deliberate: on-call engineers learn the shape once and scan it in seconds.

Use cases

UC-1 · Symptom triage — "EFT declines spiking with code 91"
"code 91" and "eft" hit curated keywords (+8), "declines" hits the title (+2) → SOP-PAY-001 at high confidence. Answer: bank-side vs link-side isolation, stand-in floor limits, pool recycle; mitigation adds 5s p95 latency alerting. Data signal shows ~250 Payments & EFT records with the Gauteng hotspot.
UC-2 · Record lookup — "what happened with INC0010000"
Regex detects the ID → record found (4G failover, Store Network & Connectivity). Its summary text is folded into scoring → SOP-NET-001. The reply opens with the record card (P3 · In Progress · Gauteng) before RCA/Fix/Mitigation.
UC-3 · Heat-wall pattern — drill into POS & Tills × Dec 25, press "Ask agent about this pattern"
The drawer composes the question automatically ("We are seeing 48 POS & Tills issues clustered in Dec 25…"). Category words route to SOP-POS-001; the trend chip quantifies the festive spike — heat wall observation to action plan in two taps.
UC-4 · Proactive alert (real-time) — P1 lands in the feed, user presses "Diagnose with agent"
The alert injects its ID, priority and summary into the query. The same pipeline runs — proactive detection and reactive diagnosis share one engine, so answers are identical either way.
UC-5 · No match — "printer in HR office jammed"
Best score falls under the threshold → the agent declines and suggests naming a retail service area or pasting an ID. No fabricated RCA.

Upgrade path

The scorer's output (top-k SOPs + data signal) is exactly the retrieval context an LLM would want. In production the local engine stays as the deterministic fallback, and a model layer composes richer narratives over the same matches — same contract, same UI.
