Enterprise upgrade path

01 Bringing an LLM into the picture

The local retriever already produces exactly what an LLM needs as grounding: the top-matched SOPs plus a live data signal from your dumps. The upgrade slots a model in at the Answer Composer — nothing upstream changes, and the deterministic engine stays as the always-available fallback.

Integration optionHow it plugs inBest when
Direct API (Anthropic / OpenAI)Backend proxies the composer call: query + top-3 SOPs + data signal → grounded narrative answer.Fastest path; POCs and mid-size estates.
Cloud-hosted (AWS Bedrock / Azure AI Foundry / GCP Vertex)Same contract, model runs inside your cloud tenancy with VPC endpoints and existing IAM.Data-residency and procurement alignment with your cloud ELA.
ITSM-native AI (ServiceNow Now Assist, Atlassian Intelligence)Replace the agent panel with platform AI; keep the heat wall as the analytics layer.Licences already owned; single-platform estates.
Self-hosted open model (Llama / Mistral on GPU)Backend serves the model; identical composer contract.Strict data controls; high query volumes where per-token cost dominates.

Two-stage pattern recommended: Stage A — LLM composes over the existing keyword retrieval (weeks, low risk). Stage B — replace keyword scoring with embeddings + vector search (RAG) once volumes justify it; the answer contract (RCA → Fix now → Mitigation → data signal) never changes.

AI LLM — RAG & vector search

01 Vector index — implemented, running in this page

The knowledge bank is chunked (each SOP splits into symptom, RCA, fix and mitigation chunks), every chunk is embedded into a 256-dimension vector using hashed TF-IDF, and stored in an in-memory vector index searched by cosine similarity. This is a genuine vector-retrieval pipeline — lexical embeddings in-browser, with neural embeddings as the production drop-in (§04).

04 Using it in the agent + production swap

The agent's engine badge is now a toggle: Keyword retrieval ↔ Vector RAG. In RAG mode the answer is grounded in retrieved chunks (shown with their similarity scores under each reply), while the RCA → Fix now → Mitigation contract stays identical — so you can A/B the two engines on the same question live.

StageIn this page (today)Production swap
ChunkingSOP → symptom / RCA / fix / mitigation chunksSame strategy over Confluence & SharePoint pages, 300–500 token chunks with overlap
EmbeddingsHashed TF-IDF, 256-dim, L2-normalised (lexical)Neural models: Voyage, OpenAI text-embedding-3, Amazon Titan, Cohere (semantic — handles synonyms & paraphrase)
Vector DBIn-memory Float32 array, exact cosine scanpgvector, OpenSearch k-NN, Pinecone, Chroma, Azure AI Search — ANN indexes (HNSW) for millions of chunks
GenerationDeterministic composer over retrieved chunksLLM composes over the same top-k context — chunks become citations

Honest limits of the in-page version: lexical embeddings match shared vocabulary, not meaning — "card machine broken" won't reach "EFT terminal fault" unless the words overlap. That semantic gap is exactly what the neural-embedding swap closes; the pipeline, index shape and answer contract are already in place.
